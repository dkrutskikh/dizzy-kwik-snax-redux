# KwikSnax Redux — UX

## Экраны

Интерфейс состоит из минимального количества экранов и обеспечивает прямой переход к игровому процессу.

Основные экраны:

- главный экран
- экран выбора режима
- игровой экран
- экран паузы
- экран завершения уровня
- экран поражения
- экран результатов

Требования:

- переходы между экранами — не более 200 ms или 12 кадров (при 60 FPS), в зависимости от того, что применимо
- анимация загрузки — не более 500 ms
- запуск игры: не более 3 действий игрока и 2 с суммарно до начала геймплея
- перезапуск уровня — не более 100 ms

На системах с двумя экранами:

- основной экран используется для игрового поля
- дополнительный экран используется для информации, меню или статистики

Меню должно управляться одинаково во всех режимах.


## HUD (индикаторы на экране)

HUD должен быть минимальным и не мешать игровому процессу.

Основные правила:

- игровое поле имеет приоритет над интерфейсом
- HUD читается с первого взгляда
- важная информация должна быть всегда видна
- элементы HUD не должны перекрывать игровые объекты

HUD отображает (каждое значение обновляется при изменении источника; не более одного шага симуляции запаздывания — **шаг симуляции** — это единое обновление физики и управления за кадр, как в [Design Pillars](02_design_pillars.ru.md)):

- очки: целое, суммарный счёт сессии; обновление при каждом начислении
- жизни: целое; обновление при потере или получении жизни
- номер уровня: целое; обновление при смене уровня
- бонусы: активные иконки или счётчики бонусов; обновление при старте/окончании бонуса
- число врагов: целое, оставшиеся враги на уровне; обновление при появлении или удалении врага
- счётчик комбо: целое `combo_count` и индикатор окна комбо (оставшееся время окна в секундах или полоса за N с по [Gameplay](03_gameplay.ru.md)); обновление при квалифицирующих действиях и каждый тик, пока окно активно
- таймер уровня: оставшиеся секунды целым числом или mm:ss; обратный отсчёт в геймплее, пауза при паузе игры, сброс при перезапуске уровня
- состояние игрока: здоровье или иконка состояния по данным дизайна; обновление при смене состояния
- состояние второго игрока: те же поля, что у первого, в кооперативе; обновление независимо по игрокам

На двухэкранных системах HUD выносится на второй экран.

Изменения состояния сопровождаются короткой анимацией (≤200 ms), звуковым сигналом и изменением цвета. Длительность анимации обратной связи задана для проверки (≤200 ms или ≤12 кадров).

## Читабельность

Игровой процесс должен оставаться читаемым при высокой скорости.

Основные требования:

- объекты должны отличаться по форме
- объекты должны отличаться по цвету
- опасные объекты должны быть заметны сразу
- игрок находит своего персонажа по силуэту и цвету
- фон не должен мешать игровым объектам

Правила читабельности:

- не использовать мелкие детали
- не использовать близкие по яркости цвета
- не перегружать экран объектами
- избегать мигающих элементов кроме заданных случаев обратной связи

Все важные события должны быть заметны без остановки движения.

## Обучение

Игра должна быть понятна без длинных инструкций.

Основные принципы:

- первые уровни обучают через игровой процесс
- новые механики вводятся постепенно
- игрок может вывести правила без текста
- ошибки отображаются явной обратной связью

Используются:

- короткие подсказки
- простые иконки
- визуальные сигналы
- демонстрация механики на уровне

Нельзя:

- показывать длинные тексты
- останавливать игру надолго
- требовать чтения перед началом игры

Игрок может начать играть в рамках бюджета запуска (≤3 действия, ≤2 с) и вывести правила в процессе. Пороги проверяемы по ms, кадрам и числу действий.
